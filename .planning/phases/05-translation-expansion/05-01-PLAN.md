  <action>
  创建 scripts/i18n/commands/validate.py，实现 `cmd_validate()` 函数：

  1. 加载翻译映射表（通过 load_translation_map）
  2. 获取 translations 字典（已展平的 {en: zh}）
  3. 执行三类质量检查：

  **检查 1: 中英混杂检测 (chinese_english_mixing)**
  - 遍历每个翻译值（中文）
  - 使用正则匹配翻译值中的连续英文字母序列（长度 >= 3，排除常见缩写如 "API", "CLI", "MCP", "URL", "SSH", "Git", "SDK", "JSON"）
  - 如果翻译值包含 3 个或以上连续英文字母（不在白名单中），标记为问题
  - 白名单包含：Claude, Code, MCP, API, CLI, URL, SSH, Git, SDK, JSON, HTML, CSS, Node, npm, LSP, SSE, IDE, PDF, DXT, NaN, OAuth, ID
  - 问题格式：{"type": "chinese_english_mixing", "key": "英文原文", "value": "翻译值", "detail": "翻译中包含未翻译的英文片段: 'Loading'"}

  **检查 2: 同义不一致检测 (synonym_inconsistency)**
  - 构建反向映射：中文翻译值 -> 对应的英文原文列表
  - 对于同一个英文原文，检查是否存在多个不同的翻译值（只比较字符串值，忽略 v6 contexts 中的变体）
  - 注意：这里不需要反向映射。而是对于同一个英文原文 key，如果它在 raw_translations 中是 v6 格式（有 contexts），则其 contexts 中的不同翻译是合理的（上下文感知），不算不一致。但如果同一个英文原文的"全局默认"翻译在不同条目中出现差异，或者同一上下文标签下有不同翻译，才标记
  - 简化实现：由于当前所有 1,301 条都是纯字符串格式（v4），同义不一致在这个阶段不会出现。但预留检查逻辑：对于 raw_translations 中同一 key，如果 "zh" 全局默认值与 contexts 中某个值相同，这是合理的。但如果同一个 key 出现在 translations 中两次（不可能，因为是 dict），跳过
  - 实际检查：对于 raw_translations 中 v6 格式的条目，检查同一上下文标签下的翻译是否与 "zh" 默认值不一致（这是正常的上下文感知），但检查 contexts 字典内是否有重复的标签值（异常情况）
  - 最终简化：检查所有翻译值中是否存在英文原文相同但翻译值不同的情况。对于 v4 格式（当前全部），这不可能（dict key 唯一）。对于 v6 格式，检查 contexts 中所有翻译是否合理
  - 最简实现：遍历 raw_translations，对每个 v6 格式条目，收集其所有翻译值（zh + contexts 中的所有值），如果两个翻译值的含义应该相同但表达完全不同，标记。但"含义相同"无法自动判断。
  - **最终决策：** 检查反向映射 — 中文翻译 -> 英文原文列表。如果同一个中文翻译对应多个完全不同的英文原文（如 "未找到" 同时翻译 "Not found" 和 "Page missing"），这是正常的。反过来检查：同一个英文原文在 translations 中只有一个值，但在 raw_translations 的 contexts 中有多个不同翻译，如果这些翻译之间的编辑距离过大（完全不相关），可能有问题。
  - **实际实现：** 简单遍历 raw_translations，对于每个 v6 条目（有 contexts），检查其 contexts 中所有值是否都是有效中文字符串。同时，对于所有翻译条目，检查翻译值是否为空字符串。

  **重新设计检查 2（更实用）：** 检测"英文原文相似但翻译不一致"的情况。例如 "Not found" 和 "Not Found" 的翻译应该一致。实现：
  - 构建映射：英文原文标准化（小写+去标点）-> [(原文, 翻译), ...]
  - 如果同一标准化英文对应多个不同翻译值，标记为 synonym_inconsistency
  - 问题格式：{"type": "synonym_inconsistency", "canonical": "not found", "entries": [{"en": "Not found", "zh": "未找到"}, {"en": "Not Found", "zh": "找不到"}]}

  **检查 3: 格式占位符丢失检测 (placeholder_missing)**
  - 从英文原文中提取所有格式占位符：
    - `%s`, `%d`, `%f`, `%i`, `%o` — printf 风格
    - `{0}`, `{1}`, `{name}` — Python/JS format 风格
    - `${...}` — JS template literal
    - `{variable}` — 常见花括号变量
  - 正则提取：`r'(%[sdifo]|\$\{[^}]+\}|\{[^}]+\})'`
  - 检查翻译值中是否包含相同的占位符
  - 如果原文有占位符但翻译中没有，标记为问题
  - 问题格式：{"type": "placeholder_missing", "key": "英文原文", "value": "翻译值", "detail": "原文占位符 ['%s'] 在翻译中丢失"}

  4. 输出 JSON 报告：

  ```json
  {

    "total_entries": 1301,
    "issues_found": 15,
    "issues_by_type": {
      "chinese_english_mixing": 5,
      "synonym_inconsistency": 3,
      "placeholder_missing": 7
    },
    "issues": [...]
  }
  ```
